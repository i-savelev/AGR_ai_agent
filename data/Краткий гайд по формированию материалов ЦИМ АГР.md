# Краткий гайд по формированию материалов ЦИМ АГР

Этот гайд собран по трем источникам:

1. Формальные требования: [Требования к ЦИМ АГР](/data/Требования%20к%20ЦИМ%20АГР.md).
2. Практика текущей приемки и ответы регулятора: [экспорт чата IFC.mos](/data/ifc_mos.md).
3. Правила расчета площадей и ТЭП: [Методика определения ТЭП](/data/Методика%20определения%20ТЭП.md).

Если чат и текст требований расходятся, для фактической сдачи ориентироваться нужно на текущую практику приемки, но такие расхождения лучше отдельно фиксировать внутри проекта.

# Как проверять материалы перед сдачей

Проверку лучше строить в четыре слоя, а не ограничиваться только просмотром IFC.

## 1. Проверка состава подачи

1. Сверить состав архива с перечнем материалов, который фактически принимается: ведомость XML, IFC по разделам, TRM, ТЭП XML.
2. Проверить, что имена IFC соответствуют требованиям раздела 4.3, [таблице 4.2](/data/Требования%20к%20ЦИМ%20АГР.md) и совпадают с именами в ведомости XML по [таблице 4.1](/data/Требования%20к%20ЦИМ%20АГР.md).
3. Проверить, что подписи УКЭП или ИУЛ оформлены и лежат рядом с файлами, если этот сценарий применяется.

## 2. Проверка IFC по атрибутам и классам

Основной быстрый контур проверки:

1. Проверка классов IFC.
2. Проверка наличия обязательных `RusSet_*` и `RUS_*`.
3. Проверка типов данных атрибутов.
4. Проверка допустимых значений кодов МССК и названий зон.

Для этого в репозитории уже есть локальные IDS:

1. [ids/АГР_Общие требования.ids](/Users/ilya/Documents/GitHub/AGR_ai_agent/ids/АГР_Общие%20требования.ids:1) — основной набор проверок по АР и общим атрибутам.
2. [ids/АГР_ЦИМ ПС.ids](/Users/ilya/Documents/GitHub/AGR_ai_agent/ids/АГР_ЦИМ%20ПС.ids:1) — проверка ПС.
3. [ids/АГР_ЦИМ БиО.ids](/Users/ilya/Documents/GitHub/AGR_ai_agent/ids/АГР_ЦИМ%20БиО.ids:1) — проверка БиО.
4. [ids/Классификация_элементов_ЦИМ_МССК_5_0_1_1.ids](/Users/ilya/Documents/GitHub/AGR_ai_agent/ids/Классификация_элементов_ЦИМ_МССК_5_0_1_1.ids:1) — контроль классификации элементов по МССК.

Практически это означает:

1. АР нужно прогонять как минимум через `АГР_Общие требования.ids` и IDS по классификации МССК.
2. ПС нужно прогонять через `АГР_ЦИМ ПС.ids`.
3. БиО нужно прогонять через `АГР_ЦИМ БиО.ids`.

## 3. Проверка XML

1. Ведомость ЦИМ XML проверять на совпадение состава файлов и нейминга.
2. ТЭП XML проверять на соответствие расчетному объему подачи и логике ГЗК.
3. Для ведомости и ТЭП полезно делать не только XSD-проверку, но и смысловую сверку с IFC.

## 4. Проверка TRM

1. Проверить, что контейнер собирается по нужной структуре.
2. Проверить, что внутри лежат нужные виды, планы, разрезы, файлы разметки и связанная IFC-модель.
3. Если проект разбит по корпусам, проверить, что структура TRM соответствует этой разбивке.

## Инструменты проверки

В рабочем контуре можно использовать:

1. `xml-agr.ru` для ведомости XML и быстрой проверки нейминга.
2. `xml-agr.ru/ifc-validator.html` для экспресс-проверки IFC по IDS.
3. Локальные IDS из папки [ids](/Users/ilya/Documents/GitHub/AGR_ai_agent/ids).
4. Любой IDS-checker, который корректно понимает IFC4 и не искажает результаты по property sets.

## Ограничения проверки

1. Проверка по IDS не заменяет ручную проверку смысловой структуры модели.
2. В чате несколько раз отмечалось, что отдельные viewer- и validator-инструменты могут показывать ложные ошибки или пропускать корректные данные.
3. Для ПС сейчас есть рассогласование между формальными требованиями, IDS и текущей практикой приемки: часть IDS и модулей все еще ожидает логику `IfcSite`, а в чате уже допускается `IfcCivilElement`.
4. Поэтому критичные места нужно проверять двумя способами: по формальной IDS и по текущему приемочному сценарию.

# Состав ZIP-архива на подачу

По текущей практике из чата в архив собирают:

1. Ведомость ЦИМ АГР в формате XML.
2. ЦИМ ПС в формате IFC.
3. ЦИМ БиО в формате IFC.
4. ЦИМ АР в формате IFC, одна или несколько.
5. ЦИМ БФ в формате IFC, при наличии.
6. Контейнер TRM в формате ZIP, один или несколько.
7. ТЭП в формате XML.

Состав подтвержден в [чате IFC.mos](/data/ifc_mos.md). Формально требования точно описывают ведомость XML, IFC-модели, атрибуты и ТЭП, а обязательность TRM сейчас подтверждается именно практикой пилотов и ответами в чате.

# Ведомость ЦИМ АГР

## Что это

XML-файл с перечнем передаваемых дисциплинарных моделей ЦИМ АГР.

## Где смотреть

1. [Требования к ЦИМ АГР](/data/Требования%20к%20ЦИМ%20АГР.md), раздел 4.
2. [Таблица 4.1](/data/Требования%20к%20ЦИМ%20АГР.md) по составу ведомости.
3. [Таблица 4.2](/data/Требования%20к%20ЦИМ%20АГР.md) по полям именования файлов.
4. Раздел о юридической значимости и подписи файлов.
5. [Чат IFC.mos](/data/ifc_mos.md), сообщение про сервис `xml-agr.ru` для сборки XML по IFC.

## Как формировать

1. Сначала определить полный набор IFC-файлов, которые реально уходят в подачу.
2. Для каждого файла задать корректное имя по схеме `<Поле1>_<Поле2>_<Поле3>_<Поле4>_<Поле5>`.
3. В XML занести порядковый номер, имя файла, краткое описание и раздел проектной документации.
4. Проверить, что имена в XML полностью совпадают с именами самих IFC-файлов.
5. Проверить, что файл подписи УКЭП лежит рядом с подписываемым файлом, если подпись оформляется на уровне файла.

## Что важно учесть

1. В имени файла обязательны символ `_` как разделители, пробелы и спецсимволы запрещены.
2. Для объектов непроизводственного назначения в поле 1 используется `НН`.
3. Если раздел разбит на несколько файлов, в шифр дисциплины добавляют номер, например `АР1`.
4. Практически ведомость удобно собирать через сервис `https://xml-agr.ru/`, который также помогает проверить нейминг.

# ЦИМ ПС

## Что это

Модель проектной ситуации: границы участка, рельеф с высотами, функциональное зонирование территории.

## Где смотреть

1. [Требования к ЦИМ АГР](/data/Требования%20к%20ЦИМ%20АГР.md), раздел 5.2.
2. [Приложение 1](/data/Требования%20к%20ЦИМ%20АГР.md), таблица А.1 по участку застройки.
3. [ids/АГР_ЦИМ ПС.ids](/Users/ilya/Documents/GitHub/AGR_ai_agent/ids/АГР_ЦИМ%20ПС.ids:1).
4. [Чат IFC.mos](/data/ifc_mos.md), обсуждения по `IfcSite` и `IfcCivilElement`.

## Как формировать

1. Смоделировать участок, рельеф и функциональное зонирование.
2. Обеспечить привязку к единой системе координат проекта.
3. Для информации об участке заполнить атрибуты `RUS_LandID` и `RUS_AreaSquare`.
4. Проверить, что модель не содержит `IfcBuildingElementProxy` вместо осмысленных классов.

## Что важно учесть

1. В требованиях участок в приложении А.1 описан через `IfcSite`.
2. По текущей позиции из чата для ЦИМ ПС допустимо использовать как `IfcSite`, так и `IfcCivilElement`.
3. Если нужно сдавать прямо сейчас, безопасно считать рабочим правилом: участок и общая информация об участке держатся в `IfcSite`, а элементы функционального зонирования и рельефа можно делать так, как проходит текущую проверку в используемом конвейере.
4. Внутри проекта нужно отдельно проверить, какой вариант проходит ваш экспорт, IDS и модуль ТЭП без ручных доработок.
5. Локальный IDS для ПС уже допускает оба класса: `IfcSite` и `IfcCivilElement`, но это не гарантирует, что все внешние проверочные модули обновлены так же.

# ЦИМ БиО

## Что это

Модель благоустройства и озеленения.

## Где смотреть

1. [Требования к ЦИМ АГР](/data/Требования%20к%20ЦИМ%20АГР.md), раздел 5.2.
2. [Приложение 1](/data/Требования%20к%20ЦИМ%20АГР.md), таблица А.2 по БиО.
3. [ids/АГР_ЦИМ БиО.ids](/Users/ilya/Documents/GitHub/AGR_ai_agent/ids/АГР_ЦИМ%20БиО.ids:1).
4. [Чат IFC.mos](/data/ifc_mos.md), ответы по `RUS_Name`, покрытиям и классу `IfcCivilElement`.

## Как формировать

1. Моделировать функциональные зоны БиО классом `IfcCivilElement`.
2. Для зон, которые участвуют в текущем расчете ТЭП, заполнять `RUS_Name` строго нормативными значениями.
3. Заполнять площадь в `RUS_Area`.
4. Проверять экспорт не только визуально, но и через IFC-валидатор или сервис проверки.

## Что важно учесть

1. Формально в приложении А.2 перечислены `Газон`, твердые покрытия, детская площадка, площадка для отдыха взрослых.
2. По актуальным разъяснениям из чата после мартовского обновления ТЭП покрытия сейчас не нужны.
3. Для текущей практики приемки безопасные значения `RUS_Name`: `Газон`, `Детская площадка`, `Площадка для отдыха взрослых`.
4. Произвольные или просто "похожие по смыслу" имена считаются ошибкой.
5. Для БиО в чате подтвержден рабочий класс `IfcCivilElement`.
6. Перед сдачей обязательно прогонять модель по IDS и отдельно смотреть, не появились ли лишние значения `RUS_Name`, не предусмотренные приложением А.2 и текущей практикой ТЭП.

# ЦИМ АР

## Что это

Основная архитектурная модель, по которой проверяют объем, фасады, оси, пространства и значительную часть ТЭП.

## Где смотреть

1. [Требования к ЦИМ АГР](/data/Требования%20к%20ЦИМ%20АГР.md), раздел 5.3.
2. [Таблица 5.1](/data/Требования%20к%20ЦИМ%20АГР.md) по классификации элементов АР.
3. [Таблица 5.2](/data/Требования%20к%20ЦИМ%20АГР.md) по минимальному набору атрибутов АР.
4. [Таблица 5.3](/data/Требования%20к%20ЦИМ%20АГР.md) по аннотациям к АР.
5. [Приложение 2](/data/Требования%20к%20ЦИМ%20АГР.md), таблицы Б.0, Б.1, Б.2, Б.3.1, Б.3.2 и далее по элементам.
6. [ids/АГР_Общие требования.ids](/Users/ilya/Documents/GitHub/AGR_ai_agent/ids/АГР_Общие%20требования.ids:1).
7. [ids/Классификация_элементов_ЦИМ_МССК_5_0_1_1.ids](/Users/ilya/Documents/GitHub/AGR_ai_agent/ids/Классификация_элементов_ЦИМ_МССК_5_0_1_1.ids:1).
8. [Методика определения ТЭП](/data/Методика%20определения%20ТЭП.md), разделы про СПП в ГНС, общую площадь и нежилые площади.
9. [Чат IFC.mos](/data/ifc_mos.md), обсуждения по зонам, этажности, осям и экспорту.

## Как формировать

1. Включить в модель помещения, зоны, ограждающие конструкции, перекрытия, заполнение проемов, лестницы, пандусы, рампы и оси.
2. Для каждой дисциплинарной модели АР заполнить `RUS_FNO` значением `Жилое здание` или `Нежилое здание`.
3. Для расчета ТЭП обязательно сформировать зоны `IfcSpace` как минимум для `СПП в ГНС` и `Общая площадь`.
4. Для помещений и зон заполнять атрибуты из приложения Б, включая `RUS_Name`, `RUS_Area`, `RUS_Volume`, `RUS_Object`, `RUS_Object_Sub`.
5. Если АР разбита на несколько файлов, общий объем наземной и подземной частей держать в `АР1`.
6. Перед сдачей прогнать АР по IDS на классы, обязательные атрибуты и коды МССК, а затем вручную проверить критичные зоны `IfcSpace` в самом IFC.

## Что важно учесть

1. Требования прямо говорят, что без корректных `IfcSpace` по СПП в ГНС и общей площади портал не сможет корректно собрать ТЭП.
2. Наименования зон в `RUS_Name` должны быть однозначными, например `СПП в ГНС` и `Общая площадь`.
3. Отдельные зоны для террас, стилобата, неэксплуатируемой кровли, чердаков и технических пространств обычно не требуются, если это специально не запрошено.
4. По методике ТЭП именно АР-модель определяет значительную часть расчетов по СПП, общей площади, этажности и нежилым площадям.
5. Если в модели отсутствуют сущности основных этажей, потом начнут расходиться этажность, оси и ТЭП.
6. В чате отмечали риск ложных ошибок IDS, если один и тот же параметр выгружен и как типовой, и как экземплярный. Перед массовой выгрузкой это лучше проверить на тестовом IFC.

# ЦИМ БФ

## Что это

Вспомогательный или базовый IFC-файл, который подается при необходимости.

## Где смотреть

1. [Чат IFC.mos](/data/ifc_mos.md), ответ по базовому файлу для выгрузки осей.
2. [Требования к ЦИМ АГР](/data/Требования%20к%20ЦИМ%20АГР.md), раздел 5.3 по обязательности координатной разбивки осей в ЦИМ АР.

## Как формировать

1. Делать только если без него нельзя передать обязательные сведения, чаще всего оси.
2. Использовать как технический файл-помощник, если основной АР-экспорт не выдает нужные оси корректно.

## Что важно учесть

1. По практике чата ЦИМ БФ особенно нужен тем, кто работает в Revit до 2022 версии.
2. Основной сценарий использования: выгрузить оси, которые не попадают в основной IFC корректно.
3. Если основной АР-файл уже содержит оси и проходит проверку, отдельный БФ может не понадобиться.

# TRM-контейнер

## Что это

Контейнер аннотаций и сопутствующей разметки, который по текущей практике обязателен к подаче.

## Где смотреть

1. [Требования к ЦИМ АГР](/data/Требования%20к%20ЦИМ%20АГР.md), раздел 6.1 по аннотациям и BCF.
2. [Таблица 5.3](/data/Требования%20к%20ЦИМ%20АГР.md) по аннотациям к ЦИМ АР.
3. [Чат IFC.mos](/data/ifc_mos.md), ответы про TRM как доработанный BCF и про обязательность контейнера на практике.

## Как формировать

1. Подготовить аннотации, которые нужны для чтения проектного решения: планы, разрезы, фасады, привязки, размеры, номера помещений и площади.
2. Собирать их в TRM-контейнер через используемый плагин или сервис Якоря.
3. Для нескольких корпусов ориентироваться на раздельную сборку TRM по корпусам, если IFC также раздроблены.

## Что важно учесть

1. Формальные требования еще описывают BCF-логику, но в текущей практике регулятор уже требует TRM.
2. По чату TRM это доработанный BCF.
3. Функционал выгрузки TRM доступен без авторизации; авторизация нужна для прямой отправки в сервис проверки.
4. По практике чата если проект разбит на несколько корпусов и несколько IFC, TRM тоже делают раздельно.
5. TRM по смыслу становится заменой пакету чертежных аннотаций, поэтому его нужно рассматривать как обязательную часть сдачи, а не как дополнение "если успеем".
6. Автоматической локальной проверки TRM в репозитории сейчас нет, поэтому для него особенно важна ручная сборочная проверка.

# ТЭП XML

## Что это

XML-файл технико-экономических показателей, который формируется не "сам по себе", а как результат согласованной логики расчета по АР, ПС и БиО.

## Где смотреть

1. [Требования к ЦИМ АГР](/data/Требования%20к%20ЦИМ%20АГР.md), раздел 7.
2. [Таблица 7.1](/data/Требования%20к%20ЦИМ%20АГР.md) по составу обязательных ТЭП.
3. XML-схема ТЭП и пример заполнения в конце [Требований к ЦИМ АГР](/data/Требования%20к%20ЦИМ%20АГР.md).
4. [Методика определения ТЭП](/data/Методика%20определения%20ТЭП.md), разделы по СПП в ГНС, нежилой площади, озеленению и площадкам.
5. [Чат IFC.mos](/data/ifc_mos.md), практические ответы по составу ТЭП и по тому, как считать объем подачи.

## Как формировать

1. Сначала определить, какой именно объем АГР вы сдаете: одно здание, несколько корпусов, один комплекс или несколько отдельных подач.
2. Подготовить АР-модель так, чтобы из нее автоматически собирались `СПП в ГНС`, `Общая площадь`, высота, этажность и другие ключевые показатели.
3. Подготовить ПС и БиО так, чтобы из них корректно читались площадь участка, озеленение, детские площадки и площадки отдыха взрослых.
4. Сверить правила подсчета не только с моделью, но и с тем, что было согласовано в ГЗК.
5. Сформировать XML ТЭП через автоматизированный инструмент и проверить расхождения до подачи.

## Что важно учесть

1. Ключевой практический принцип из чата: ТЭП подаются по тому объему, с которым вы заходите в АГР. Если в одном АГР подаются два здания, ТЭП суммируются; если на каждое здание отдельное АГР, ТЭП формируются раздельно.
2. По методике ТЭП правила расчета СПП в ГНС зависят от актуального режима подсчета и трактовки проемов, антресолей, вторых светов, кровель и цокольных этажей.
3. Поэтому ТЭП нельзя делать "по аналогии" или вручную из таблицы. Нужно сначала правильно смоделировать зоны и площади.
4. По текущей практике для БиО в ТЭП значимы `Газон`, `Детская площадка`, `Площадка для отдыха взрослых`; покрытия сейчас не являются обязательной частью расчета.
5. Если ГЗК считался по особым правилам, модель должна воспроизводить именно эту согласованную логику, иначе XML формально соберется, но проверку по смыслу не пройдет.
6. Проверять ТЭП нужно не только по XML, но и по источнику данных в IFC: ошибка в зоне `IfcSpace` или `RUS_Name` почти всегда приводит к корректному по структуре, но неверному по смыслу XML.

# Минимальная последовательность работы

1. Определить состав подачи: какие именно IFC, сколько корпусов, один ли это объем АГР.
2. Сразу согласовать схему именования файлов и разбиение моделей.
3. Собрать и проверить ЦИМ ПС и ЦИМ БиО.
4. Собрать и проверить ЦИМ АР, включая оси, `IfcSpace` и атрибуты `RUS_*`.
5. При необходимости сформировать ЦИМ БФ для осей.
6. Сформировать ведомость XML.
7. Сформировать ТЭП XML и сверить с ГЗК.
8. Сформировать TRM-контейнеры.
9. Перед упаковкой ZIP прогнать IFC и XML на техническую проверку.

# На что смотреть в первую очередь перед сдачей

1. Проходит ли нейминг файлов.
2. Есть ли в АР корректные `IfcSpace` для `СПП в ГНС` и `Общая площадь`.
3. Заполнены ли `RUS_FNO`, `RUS_Name`, `RUS_Area`, `RUS_LandID`, `RUS_AreaSquare` там, где они обязательны.
4. Не остались ли в критичных местах произвольные классы или `IfcBuildingElementProxy`.
5. Совпадает ли логика ТЭП с тем, что было принято в ГЗК.
6. Сформированы ли TRM-контейнеры по фактической структуре подачи.
7. Прогнаны ли IFC через соответствующие IDS из папки [ids](/Users/ilya/Documents/GitHub/AGR_ai_agent/ids).
8. Проверены ли вручную места, где текущая практика чата уже расходится с формальными IDS или текстом требований.
